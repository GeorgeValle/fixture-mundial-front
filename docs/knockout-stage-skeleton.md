# Knockout Stage Skeleton — FIFA World Cup 2026

This document defines the initial knockout stage skeleton for the FIFA World Cup 2026 fixture frontend.

## Purpose

This file is a temporary structural reference for the Knockout Stage page.

It should be used only to render placeholder rounds, dates, match numbers, stadiums, and expected qualification labels until the real knockout matches are available from the backend/database.

## Important Rules

- Do not treat this file as final match result data.
- Do not invent qualified teams.
- Do not invent scores.
- Do not invent penalties.
- Do not replace backend data with this file when real data exists.
- The frontend should prioritize matches received from the database/backend.
- This skeleton is only a fallback/reference to build the initial bracket UI.
- Team names such as `Winner Match 74`, `Runner-up Group A`, or `3rd Group A/B/C/D/F` are placeholders.
- Once real teams qualify, the database/backend must provide the resolved teams and match results.

## Data Priority

The Knockout Stage page should follow this priority:

1. Use real knockout match data from the backend/database when available.
2. If real knockout data is incomplete or unavailable, render this skeleton as placeholder data.
3. Clearly indicate in the UI that unresolved matches are pending official qualification/results.

# Implementation Notes

The frontend should use this document as a planning/reference source for the initial Knockout Stage UI.

Recommended UI behavior:

- Show each round as a separate section.
- Show each match number.
- Show the scheduled date.
- Show the stadium.
- Show placeholder team labels when real teams are not known.
- Show a clear visual badge such as `Pending official data`, `Placeholder`, or `To be confirmed`.
- Replace placeholders with real backend data when available.
- Show regular score only when the backend provides it.
- Show penalties only when the backend provides them.
- Avoid rendering placeholder labels in a way that looks like confirmed teams.

Recommended data behavior:

- Backend/database data must be the source of truth.
- This skeleton should not override backend results.
- If a backend match exists with the same match number, use the backend match.
- If a backend match is missing, use the skeleton item as a placeholder.
